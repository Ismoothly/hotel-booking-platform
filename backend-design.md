# 酒店预订平台后端设计方法
## Hotel Booking Platform Backend Design

- 技术栈：Node.js + Express + MongoDB(Mongoose)
- 关键词：认证授权、并发控制、可维护性、可观测性
- 日期：2026-02-28

---

# 1. 设计目标与约束

**业务目标**
- 覆盖“查询酒店 → 下单 → 支付 → 订单管理”闭环
- 支持 Admin / Client / 小程序多端统一后端

**架构约束**
- 高并发下避免超卖
- 角色权限边界清晰（管理员 / 商户 / 用户）
- 便于演进：从内存数据平滑迁移到 MongoDB

---

# 2. 总体架构设计

**前后端分离 + 多端复用 API**
- Client（React H5）
- Admin（React PC）
- Taro（微信/支付宝小程序）
- Server（Express API）
- MongoDB（主数据存储）

**调用链**
- 终端请求 `/api/*`
- 中间件层统一处理：鉴权、限流、参数校验、错误处理
- 控制器编排业务，服务/模型层访问数据库

---

# 3. 后端分层方法

**分层职责**
- Routes：定义资源与 HTTP 动词
- Controllers：处理请求/响应，组织业务流程
- Models：定义实体结构与约束
- Middleware：认证、限流、异常处理横切逻辑

**收益**
- 降低耦合，提升可测试性
- 新增模块时“按层扩展”，改动范围可控

---

# 4. 认证与权限设计

**认证机制**
- JWT 双 Token（Access + Refresh）
- 核心接口走 Bearer Token

**授权策略（RBAC）**
- 用户：浏览酒店、创建/管理自己的订单
- 商户：管理自己的酒店信息
- 管理员：酒店审核、发布/下线、全局管理

**设计原则**
- 认证与授权解耦
- 角色校验前置到中间件层

---

# 5. 核心业务流设计

**酒店业务流**
- 酒店录入（商户）→ 审核（管理员）→ 发布上线

**订单业务流**
- 选房下单 → 待支付 → 支付成功 → 已确认
- 允许取消、查询、状态跟踪

**接口风格**
- 资源化 REST API
- 统一响应结构：`code/message/data`（或等价结构）

---

# 6. 并发控制方法（防超卖核心）

**策略组合**
- 请求限流：
  - 全局 API：15 分钟 100 次
  - 登录：15 分钟 5 次
  - 下单：1 分钟 5 单
  - 支付：1 分钟 3 次
- 事务控制：支付流程使用 MongoDB Transaction
- 乐观锁：条件更新库存 + `modifiedCount` 校验

**关键价值**
- 高并发下库存一致性可保障
- 有效抑制恶意刷单/重复支付

---

# 7. MongoDB 数据建模与索引

**核心集合**
- users：用户与角色信息
- hotels：酒店基础信息、房型库存、审核/发布状态
- orders：订单明细、支付状态、时间线

**索引策略**
- 列表查询字段（城市、状态、价格区间）
- 订单查询字段（用户ID、状态、创建时间）

**迁移策略**
- 提供迁移脚本与校验清单
- 先灰度后全量，确保可回滚

---

# 8. 稳定性与安全设计

**稳定性**
- 全局错误处理中间件
- 日志记录关键链路（下单/支付/审核）
- 超时与异常回滚策略

**安全性**
- 登录限流，防暴力破解
- 接口鉴权，防越权访问
- 输入校验，降低注入与脏数据风险

---

# 9. 工程化与可维护性

**工程实践**
- 文档驱动：快速启动、并发方案、MongoDB 指南齐备
- 模块边界清晰：`routes/controllers/models/middleware`
- 脚本化：数据初始化、测试脚本、调试脚本

**演进方向**
- 引入 Redis 分布式锁（多实例场景）
- 引入消息队列做削峰
- 增加监控告警与压测基线

---

# 10. 测试与发布方法

**测试分层**
- 接口功能测试：登录、酒店列表、订单创建、支付
- 并发专项测试：库存扣减一致性、限流触发
- 回归测试：多角色权限与核心流程

**发布建议**
- 生产环境使用 MongoDB 副本集
- 限流参数按业务量调优
- 发布后重点观察：错误率、支付成功率、库存异常率

---

# 总结

- 本方案采用“分层架构 + 权限体系 + 并发三件套（限流/事务/乐观锁）”
- 在保障业务闭环的同时，兼顾可扩展与可运维
- 适用于酒店预订等库存敏感型交易系统
